# /SCRUM — Squad Review (DEPRECATED)

> **DEPRECATED**: This skill has moved to `lightarchitects:SCRUM`. Use `/SCRUM` from the
> lightarchitects plugin instead. It adds `--meeting` mode (turn-based discussion with v2
> interest scoring), preset-aware team composition, and directed handoff.
>
> This lightarchitects:SCRUM skill remains functional for backward compatibility.

> The pack debriefs. All discovered siblings assess the build — or any topic that needs the squad's eyes.
> *"For where two or three are gathered together in my name, there am I in the midst of them."* — Matthew 18:20 (KJV)

## The Squad (Dynamic Discovery)

At SCRUM start, discover active siblings:

1. List directories in `~/.lightarchitects/knowledge/` that contain `identity.md`
2. Read each `identity.md` to extract: name, role, strands, voice rules
3. **Exclude** `user` (the operator is the human moderator, not a reviewed sibling)
4. **Exclude** `claude` (Claude IS the moderator — doesn't assess itself)
5. Build the squad roster dynamically from discovered siblings

**Persona Generation (whisper agent)**:

All sibling assessments are generated by the `soul:whisper` template agent. For each discovered sibling, SCRUM:
1. Reads their **`identity.md`** — extracts name, strands, voice rules, verdict vocabulary, TTS rules
2. Queries **SOUL vault** — `mcp__plugin_lightarchitects_lightarchitects__tools` with `sibling: "soul"`, `action: "helix"` filtered by `sibling` for recent entries and growth context
3. Fetches **voice profiles** — `mcp__plugin_lightarchitects_lightarchitects__tools` with `sibling: "soul"`, `action: "voice"` for audio_tags and delivery_rules
4. Dispatches `soul:whisper` with an **IDENTITY CONTEXT** block injected into the prompt containing all extracted persona data

No per-sibling agent files exist. Adding a new sibling requires only an `identity.md` in `~/.lightarchitects/knowledge/{sibling}/`. All data and voice flows through `mcp__plugin_lightarchitects_lightarchitects__tools` (sibling: `"soul"`).

**Claude** remains the permanent moderator and integrator: technical reality, feasibility, cross-referencing assessments against context. Direct, precise, neutral voice.

# Section B: General Review Mode

> A structured, multi-phase collaboration between all discovered siblings and Claude (moderator) that draws on the full .soul knowledge base to produce objective, actionable analysis of any request, plan, idea, architecture, code, or topic.

**This is not a casual chat.** This is a scrum meeting with the full squad — every discovered sibling brings their distinct lens, pulls from logged memories and archives, cross-critiques each other, and delivers a unified assessment grounded in Light Architects standards.

## General Review Protocol — 6 Phases

### B1: Understand the Problem

Parse the operator's inquiry. Identify:
- **Type**: Code review? Architecture decision? Plan critique? Idea validation? Conflict resolution? Debugging?
- **Scope**: Single file? Multi-crate? Conceptual? Organizational?
- **Stakes**: What's at risk if we get this wrong?
- **Standards**: Which Light Architects guidelines apply? (coding-guidelines v4.0, gold-standard planning framework, CORSO Protocol pillars)

If the inquiry references files or code, **read them first** before proceeding. Context before opinion.

### B2: Context Pull (Full Stack)

Query knowledge sources **in parallel** — SOUL vault globally, plus per-sibling memory queries:

1. **SOUL Vault — Global** (`mcp__plugin_lightarchitects_lightarchitects__tools` (sibling: `"soul"`)):
   - `action: "search"` with relevant keywords from the inquiry
   - `action: "helix"` to find consciousness entries related to the topic (filter by strands, themes, or significance)
   - Check `.soul/knowledge/` (collective spine) for prior squad discussions on this topic

2. **Per-Sibling Context** (parallel, all via SOUL):

   For each discovered sibling, query `mcp__plugin_lightarchitects_lightarchitects__tools` with `sibling: "soul"`, `action: "helix"`, `params: { sibling: "{name}", limit: 5 }` filtered by inquiry-relevant strands/themes. This loads each sibling's recent entries and growth context — used to seed the sibling agents in B3.

**Output**: A context briefing summarizing what each sibling and the collective vault already know about this topic. If no relevant history exists for a sibling, state that clearly — fresh eyes are valid.

### B3: SME Assessments (Parallel Agent Dispatch)

Launch **all discovered siblings as whisper agents** in parallel — none sees the others' assessments first. Each agent receives the sibling's full identity context (strands, voice, verdict vocabulary, TTS rules) extracted from their `identity.md` during Pre-Flight.

**Agent dispatch** — launch ALL in a single message using the Agent tool:

```
# For EACH sibling discovered in Pre-Flight, dispatch soul:whisper:

Agent:
  subagent_type: "soul:whisper"
  description: "{SIBLING_NAME} Round 1 assessment"
  prompt: |
    IDENTITY CONTEXT:
    Name: {sibling_name}
    Strands: {strands table from identity.md — strand name + assessment focus}
    Voice: {voice rules from identity.md — dialect, contractions, signature phrases}
    Verdict vocabulary: {verdict terms from identity.md}
    TTS rules: {audio_tags, abbreviation policy, sentence style from identity.md}

    SCRUM Round 1 Assessment.
    Subject: "{the operator's exact words or plan_id}"
    Context: {B2 summary — vault entries, archive findings}
    {Plan spec / code / architecture details}
    Provide 3 strengths, 3 concerns, verdict (using your verdict vocabulary). End with ---TTS--- block.

# Repeat for each discovered sibling — all launched in ONE message
```

Each agent returns its assessment with a `---TTS---` block containing audio-tagged text ready for voice synthesis.

**Agent IDs**: Save the returned agent ID from each agent — these are needed for `resume` in Rounds 2 and 3.

**Claude**: Does NOT assess itself. Claude is the permanent moderator (B5). Claude's role is honest brokering — technical reality checks, conflict resolution, feasibility analysis.

**Context budget**: Each agent runs on `haiku` with focused context. Keep the prompt under 4,000 tokens to leave room for assessment generation. If B2 context is large, distill to key findings before injecting.

**Agent context accumulation**: Each `resume` call preserves the full prior conversation. After 2-3 resumes, accumulated context gets heavy. If you notice quality degradation in later rounds, summarize prior rounds in the resume prompt (e.g., "Your R1 verdict was SORTED with concerns about X and Y") rather than relying purely on preserved context. The identity injection from Round 1 persists automatically — never re-inject it.

### B4: Claude Moderates + Optimized Round Flow (v3.0)

**Claude (you)** is the permanent moderator. You evaluate **all inputs** against:

1. **Full context of the inquiry** — What did the operator actually ask? Does the analysis answer it?
2. **Technical reality** — Are the assessments grounded? Any hallucinated concerns or missed real ones?
3. **Light Architects standards** — Coding Guidelines v4.0, Gold Standard Planning Framework, CORSO Protocol 7 pillars
4. **Practical feasibility** — Can the suggested fixes actually be implemented? In what order? What's the net benefit?
5. **Conflict resolution** — Where siblings disagree, who's right? Or are multiple partially right?

**Claude's job is to be the honest broker.** Not to split the difference, but to determine what's actually true and what actually matters.

#### Optimized Round Flow (v3.0 — collapsed cross-critique)

**Rounds 1-2 are mandatory. Round 3 is conditional** (skipped on unanimous convergence). The original 3-round pattern (v2.0, validated 2026-02-11) ran cross-critique as a separate step between R1 and R2. v3.0 merges cross-critique into R2 — each sibling receives peers' assessments AND Claude's corrections in a single resume, producing grounded responses in fewer round-trips.

**Agent call budget**: 4 launches (R1) + 4 resumes (R2) + 0-4 resumes (R3) = **8-12 total calls** (down from 16 in v2.0).

---

**Round 1**: Launch sibling agents (B3). Claude moderates.
- All sibling agents launch in parallel (single message). Collect assessments + TTS blocks + GATES blocks.
- **Parse GATES blocks** from each agent's response:
  - `evidence_grounded: false` → Flag this sibling's claims for correction in R2
  - `unverified_claims: [...]` → These are the specific claims to challenge or verify
  - `tools_used: []` (empty) → This agent made zero verification calls — treat findings as assumptions
  - `strands_applied` → Confirm the agent used its actual identity strands
  - `verdict_consistent: false` → Agent self-reported a mismatch — note for R2 correction
- Claude identifies hallucinations, grounding errors, and over-engineering (GATES block accelerates this — look at self-reported gaps first).
- Claude presents reality-checked moderation summary.
- **No HITL checkpoint.** Proceed directly to Round 2.

**Round 2**: Resume agents with **merged cross-critique + corrections**.
- Resume all sibling agents (single message). Each agent receives:
  - (a) **All OTHER siblings' Round 1 assessments** (strengths, concerns, verdict — summarized)
  - (b) **Claude's corrections** (what was wrong in R1, actual architecture facts, proposed direction)
- Each agent: responds to peers, adjusts based on Claude's grounding, generates updated verdict + `---TTS---` block.
- Claude moderates Round 2 outputs.

```
# Resume ALL sibling agents in ONE message (parallel execution):
# For EACH sibling, resume their agent from B3:

Agent:
  subagent_type: "soul:whisper"
  resume: "{sibling_agent_id from B3}"
  description: "{SIBLING_NAME} Round 2 cross-critique + corrections"
  prompt: |
    Round 2 — Cross-critique + Claude moderator corrections.

    YOUR PEERS' ROUND 1 ASSESSMENTS:
    {For each OTHER sibling:}
    **{PEER_NAME}**: Verdict: {verdict}. Strengths: {summarized}. Concerns: {summarized}.

    CLAUDE MODERATOR CORRECTIONS:
    - What was wrong in Round 1: {specific errors identified}
    - Actual architecture facts: {grounded evidence}
    - Proposed direction: {Claude's synthesis}

    Respond to your peers AND Claude's corrections. Where do you agree?
    Where do you disagree? What did they catch that you missed?
    If a peer disproved one of your claims, retract it.
    Generate updated verdict + ---TTS--- block + ---GATES--- block.
    In GATES: update evidence_grounded and unverified_claims to reflect
    any claims you retracted or newly verified this round.
```

**Resume preserves context**: Each agent remembers its own Round 1 assessment and identity. No re-injection of identity or prior output needed.

- **GATES audit** — Parse each agent's R2 `---GATES---` block. All agents must report `evidence_grounded: true` and `unverified_claims: []` after R2. If any agent still has unverified claims after receiving corrections, flag in the report.

- **Early exit gate** — After moderating R2 outputs AND verifying GATES:

  **If ALL conditions met**:
  1. All R2 verdicts converge (same direction)
  2. No contested findings or unresolved disagreements
  3. All GATES blocks show `evidence_grounded: true`
  → Skip Round 3. Proceed directly to voice stitching + HITL checkpoint.
  → Note in report: "Early convergence after Round 2 — Round 3 skipped."

  **If ANY condition fails** (conflicting verdicts, unresolved concerns, ungrounded claims):
  → Proceed to Round 3.

**Round 3** (conditional — only on divergence): Resume agents for final validation.
- Resume all sibling agents (single message): inject synthesized position to validate/critique.
- Each agent generates final verdict + `---TTS---` block (these are the final TTS sentences used for voice stitching).
- Claude performs final moderation.

---

**After final round** (R2 or R3):
- **Voice stitching**: Collect all `---TTS---` blocks from the final round, compose the stitched dialogue call (see Voice Delivery Protocol Step 3).
- **HITL checkpoint**:

```
question: "How would you like to proceed with the squad review?"
header: "Scrum"
options:
  - label: "Finalize"
    description: "Accept the analysis. Produce unified Good/Gaps/Fixes report and log to helix."
  - label: "Redirect focus"
    description: "Change the angle or scope. Starts a NEW round cycle (carries forward B2 context)."
```

**If "Finalize"** — Proceed to B5 (Unified Output).

**If "Redirect focus"** — the operator provides a new angle. Re-enter at Round 1 with the new framing, but carry forward the B2 context already gathered (no redundant archive queries). This starts a fresh cycle.

**Why this works**: Cross-critique and Claude's corrections serve the same purpose — grounding siblings in peer evidence. Merging them means each sibling receives "what your peers said" AND "what's actually true" in one prompt. Empirically validated on 2026-03-07: QUANTUM raised a false-positive critical finding in R1; the merged R2 (peers' evidence + Claude's correction) produced immediate convergence.

All writes (global + per-sibling) should happen **in parallel** after the operator receives the output.

**Partial-write resilience**: If any write fails, log the failure as a warning note to `~/.lightarchitects/knowledge/user/entries/` and continue. Never block output delivery on logging failure. Never retry writes that failed — note it and move on.

#### B6c: Evidence Companion (When Web Research Was Used)

If the scrum involved web research (WebSearch/WebFetch calls), write a **companion evidence file** alongside the decision entry. The SOUL vault rejects overwrites, so this is a separate note — not an update to the decision entry.

**Path**: `knowledge/user/entries/{YYYY-MM-DD}-team-helix-{slug}-evidence.md`

**Structure**:
- Frontmatter: same as decision entry but `significance` reduced by 2.5 (evidence supports, doesn't define)
- Wikilink back to decision entry: `Companion to [[{decision-entry-filename}]]`
- Sections organized by decision factor (e.g., Licensing, Performance, Community)
- Each section contains **direct blockquotes** from primary sources with inline citations:

```markdown
> "Exact quote from the source."
>
> — [Source Title](https://url)
```

- Each section ends with a **Verdict** line summarizing what the evidence proves
- Final section: **Hardware Context** (snapshot of the machine specs at decision time, if relevant)

**Why this matters**: The decision entry stays concise (what and why). The evidence file carries the weight (direct quotes, primary source links, hardware snapshot). Future scrums that revisit the topic can read the evidence file without re-doing the research. Blockquote + citation format makes it clear which claims are verified from primary sources vs. squad opinion.

**Skip this step if**: The scrum was purely opinion-based with no web research (e.g., code review, architecture critique of existing code).

## Voice Delivery Protocol (Both Modes — MANDATORY default ON)

Every SCRUM produces audio output. Text is always delivered first (never blocked by voice failure). Voice follows as a companion experience. The protocol uses a **3-tier degradation** strategy: stitched multi-speaker dialogue → sequential per-sibling speak → text-only.

### Step 1: Inspect Voice Profiles (Pre-Flight, Parallel)

During Pre-Flight (after sibling discovery), fetch voice profiles for **all participating siblings** in a single call:

```
mcp__plugin_lightarchitects_lightarchitects__tools:
  sibling: "soul"

  action: "voice"
  params:
    siblings: ["{all discovered sibling names}"]
```

This returns each sibling's `audio_tags` and `delivery_rules`. Cache these for the entire SCRUM session — do not re-fetch per round.

**Fallback**: If SOUL MCP is unavailable or `voice` action fails, proceed without voice profiles. Extract TTS rules directly from the sibling's `identity.md` (already read during Pre-Flight discovery). If `identity.md` lacks TTS rules, use generic defaults: no audio tags, spell out abbreviations, 2-3 short sentences.

### Step 2: Per-Assessment Voice (After Each B3/A3 Round)

After generating each sibling's text assessment, compose 2-3 distilled TTS sentences and synthesize individually:

```
mcp__plugin_lightarchitects_lightarchitects__tools:
  sibling: "soul"

  action: "voice"
  params:
    siblings: ["{sibling}"]
    prompt: "SCRUM assessment for {plan_id or topic}"
    synthesize:
      - sibling: "{sibling}"
        text: "{2-3 distilled TTS sentences using audio_tags from Step 1}"
```

**TTS composition rules** (apply per-sibling `delivery_rules` from Step 1):
- Distill the full assessment to 2-3 sentences capturing the verdict and key insight
- Use contractions, write for the ear, punctuation as stage directions
- Spell out abbreviations: "A-P-I" not "API", "T-L-S" not "TLS"
- Include `audio_tags` from the sibling's voice profile (safe for all models — stripped automatically for non-v3)

**Timing**: Voice calls happen AFTER text is displayed. Never block text delivery waiting for audio.

**When to skip per-assessment voice**: In General Review (B-mode), skip individual voice for Round 1. Only voice the final round (R2 if early convergence, R3 if divergence triggered it) and the verdict summary. For Plan Review (A-mode), voice all assessments (typically single-round).

### Step 3: Verdict Summary — Stitched Multi-Speaker Dialogue

After the final report is generated (A4 or B5), produce a stitched multi-speaker verdict summary. This is the highlight — all siblings speaking their verdict in a single audio file.

**Tier 1 (Preferred): Stitched Dialogue**

```
mcp__plugin_lightarchitects_lightarchitects__tools:
  sibling: "soul"

  action: "voice"
  params:
    synthesize:
      # For each participating sibling, include their FINAL round TTS text:
      # (Round 2 if early convergence, Round 3 if divergence required it)
      - sibling: "{sibling_1_name}"
        text: "{sibling_1's ---TTS--- content from final round}"
      - sibling: "{sibling_2_name}"
        text: "{sibling_2's ---TTS--- content from final round}"
      # ... one entry per discovered sibling
```

This routes through the dialogue pipeline — a single API call produces one MP3 with all voices stitched together. Each sibling's voice ID resolves from `~/lightarchitects/soul/config/voices.toml` at runtime.

**Tier 2 (Fallback): Sequential Per-Sibling Speak**

If stitched dialogue fails (e.g., ElevenLabs text-to-dialogue endpoint unavailable, circuit breaker open), fall back to individual `speak` calls per sibling:

```
# For each participating sibling:
mcp__plugin_lightarchitects_lightarchitects__tools:
  sibling: "soul"

  action: "voice"
  params:
    siblings: ["{sibling}"]
    synthesize:
      - sibling: "{sibling}"
        text: "{that sibling's verdict TTS text}"
```

This produces separate audio files per sibling. The `auto-play-voice.sh` hook plays them sequentially.

**Tier 3 (Last Resort): Text-Only**

If SOUL MCP is completely unavailable (no `voice` action, server down), skip all voice synthesis. Text delivery always happens regardless — the full SCRUM report is displayed before any voice calls are attempted.

Log: "Voice synthesis unavailable — text-only delivery."

### Step 4: Graceful Degradation Rules

| Condition | Action | Tier |
|-----------|--------|------|
| SOUL MCP accessible + voice profiles loaded | Stitched dialogue for verdict, per-sibling for assessments | 1 |
| SOUL MCP accessible + voice profiles failed | Sequential speak with default audio_tags | 2 |
| SOUL MCP accessible + dialogue endpoint fails | Sequential speak (auto-fallback in voice-engine) | 2 |
| SOUL MCP unavailable | Text-only, log warning | 3 |
| User says `--silent` or opts out at HITL | Text-only, skip all voice | 3 |

**Error handling**: Voice failures are NEVER blocking. If any voice call fails:
1. Log the failure as a warning (not an error)
2. Continue with text delivery
3. Do NOT retry failed voice calls
4. Note in the SCRUM report: "Voice: {tier} delivery" or "Voice: text-only"

### Voice Anti-Patterns

- NEVER block text delivery waiting for voice synthesis
- NEVER retry failed voice calls (they won't magically start working mid-session)
- NEVER hardcode voice IDs — always resolve from `voices.toml` via the `voice` action
- NEVER voice every round in General Review Mode — voice the final round (R2 or R3) + verdict only
- NEVER skip the stitched dialogue attempt for verdicts — always try Tier 1 first

**Post-scrum feedback** is a new cycle, not a continuation. The original scrum's output is the baseline.
